## 牌谱设计

### 设计目标

牌谱模块只支持 `cukee` 自有牌谱格式，不支持读取或者保存天凤牌谱。

牌谱用于完整记录每一小局的牌山和事件信息。初始手牌、摸牌序列、宝牌指示牌、里宝牌指示牌等信息都应能由牌山和事件推算得到，不在牌谱中重复保存。

牌谱不负责维护当前牌局状态。当前手牌、牌河、巡目数、本场数、立直棒数、点数等状态信息后续由专门的牌局状态类型维护。

### 命名约定

仓库级命名约定以 `AGENTS.md` 为准。

`牌_T` 是公共基础类型，定义在 `src/common/` 中，由牌谱模块复用。

`牌谱_Event` 是单独的事件结构体，内部持有 `std::variant` 成员。后续可以在 `牌谱_Event` 上增加成员函数，提供事件类型判断、访问和辅助转换等操作。

### 模块位置

牌谱代码单独放在 `src/log/` 模块中，对应 `CMake` 目标为 `cukee_log`。

牌谱模块只负责牌谱内存结构、`cukee` 牌谱载荷读取、`cukee` 牌谱载荷保存和落盘编码，不负责规则推进和牌局状态维护。

依赖方向：

- `src/log/` 依赖 `src/common/` 中的公共基础类型，不依赖 `src/game/`。
- `src/game/` 后续可以依赖 `src/log/`，用于记录或者回放牌谱事件。
- `src/trainer/`、`src/analyzer/`、`src/simulator/` 后续可以依赖 `src/log/`，用于读取训练牌谱、保存模拟牌谱或者展示对战牌谱。

### 内存结构

内存牌谱不做压缩，优先保证结构清晰、访问直接和便于重放。

初步结构如下：

```cpp
struct 牌_T {
    char 门;
    uint8_t 编号;
    bool 红宝牌;
};

struct 牌谱_T {
    std::array<uint8_t, 4> player_ranks;
    std::array<std::string, 4> player_names;
    std::vector<小局_T> 小局;
};

struct 小局_T {
    uint8_t 场风;
    uint8_t 局编号;
    uint8_t 本场数;
    uint8_t 立直棒数;
    uint8_t 亲家;
    std::array<int32_t, 4> 起始点数;

    std::array<牌_T, 136> 牌山;
    std::vector<牌谱_Event> events;
};
```

`牌_T::门` 使用字符表示牌种，`m` 表示万，`p` 表示筒，`s` 表示条，`z` 表示字。

`牌_T::编号` 从 `0` 开始。对于数牌，`0..8` 表示 `1..9`；对于字牌，`0..6` 表示 7 种字牌。例如 `m 0` 表示一万。

`牌_T::红宝牌` 表示该牌是否为红宝牌。

玩家编号使用 `0..3`，固定表示东 1 局的庄家、南家、西家、北家。该编号不随当前小局的自风变化。

`牌谱_T::player_ranks` 保存 4 名玩家的玉之间段位，取值 `0..5`，分别表示雀豪 1、雀豪 2、雀豪 3、雀圣 1、雀圣 2、雀圣 3。

`牌谱_T::player_names` 保存 4 名玩家的名称。名称可以为空字符串，表示匿名。落盘时每个名称使用 1 字节长度加名称内容保存，长度最大为 `255`。

`小局_T` 记录当前小局开始时难以从本小局内部直接得到的状态，包括场风、局编号、本场数、立直棒数、亲家和起始点数。初始手牌、初始宝牌指示牌、小局结束类型和小局结束后点数不记录在 `小局_T` 中。

`小局_T::牌山` 保存完整 `136` 张牌的顺序。

### 事件结构

`牌谱_Event` 不使用虚函数和继承体系，避免大量牌谱同时存在时产生过多碎片和间接内存访问。

事件采用 `std::variant` 保存：

```cpp
struct 牌谱_Event {
    using Variant = std::variant<
        摸牌_Event_T,
        打牌_Event_T,
        副露_Event_T,
        暗杠_Event_T,
        加杠_Event_T,
        和牌_Event_T,
        流局_Event_T
    >;

    Variant data;
};
```

各类事件初步只保存事件本身需要的信息，成员函数后续根据回放、训练和分析需求再补充。

示例：

```cpp
struct 摸牌_Event_T {
    uint8_t actor;
    牌_T 牌;
};

struct 打牌_Event_T {
    uint8_t actor;
    牌_T 牌;
    bool 摸切;
    bool 立直宣告;
};

enum class 副露类型_T : uint8_t {
    吃,
    碰,
    明杠,
};

enum class 吃类型_T : uint8_t {
    上吃,
    中吃,
    下吃,
};

struct 副露_Event_T {
    uint8_t actor;
    uint8_t target;
    副露类型_T type;
    吃类型_T 吃类型;
    牌_T 牌;
    std::array<牌_T, 3> consumed;
    uint8_t consumed_count;
};

struct 暗杠_Event_T {
    uint8_t actor;
    牌_T 牌;
    bool 摸杠;
};

struct 加杠_Event_T {
    uint8_t actor;
    牌_T 牌;
    bool 摸杠;
};

struct 和牌_Event_T {
    std::array<bool, 4> 和;
    bool 自摸;
};

enum class 流局类型_T : uint8_t {
    荒牌,
    九种九牌,
    四风连打,
    四杠散了,
    四家立直,
};

struct 流局_Event_T {
    流局类型_T type;
};
```

`副露_Event_T` 覆盖吃、碰和明杠，`consumed_count` 对应 `2`、`2`、`3`。

`副露_Event_T::target` 表示被副露牌的来源玩家，也就是打出这张牌的玩家。吃的时候 `target` 必然是上家，但仍然保留该字段以统一事件结构；碰和明杠时 `target` 用于区分这张牌来自哪一家。

`副露_Event_T::吃类型` 表示上吃、中吃、下吃。事件不是吃时该字段无意义。

暗杠和加杠是轮到自己行动时发生的事件，因此不并入 `副露_Event_T`。

`暗杠_Event_T` 只记录 1 张牌，不记录 4 张实体牌。暗杠必然是同一牌种 4 张牌，如果是 5，则必然包含红五，`牌_T::红宝牌` 字段对暗杠无意义。

`暗杠_Event_T::摸杠` 和 `加杠_Event_T::摸杠` 表示是否使用刚摸上来的牌完成暗杠或者加杠，语义类似打牌事件中的 `摸切`。

`和牌_Event_T::和` 使用 4 个布尔值表示 4 名玩家是否选择和牌，具体结算结果后续由牌局模拟器计算。

`和牌_Event_T::自摸` 表示该次和牌是否为自摸。摸牌事件落盘时不编码，读取时需要根据该字段区分荣和与自摸，并在自摸前自动补回摸牌事件。

### 文件格式方向

`cukee` 完整牌谱文件格式后续单独细化。当前已先实现内存对象的二进制载荷级 `save` 和 `load`，完整文件头、版本号、多牌谱文件封装和边界校验后续再设计。

当前 `牌谱_T` 载荷顺序为：4 名玩家段位、4 名玩家名称、小局数量、小局载荷序列。

当前只确定原则：

- 文件保存时再做压缩，内存结构不为压缩牺牲可读性。
- 完整文件格式需要保存每一小局的牌山和事件信息。
- 完整文件格式需要包含版本号，便于后续兼容升级。
- 事件落盘可以使用比内存结构更紧凑的二进制编码。
- 压缩算法后续再选，优先考虑解压速度、压缩率和实现复杂度的平衡。
